The Problem

You have multiple GitHub accounts, and you want to git to automatically push/pull to the correct GitHub account for each local project.

Current Solution

In this article I set things up so that:

  • Projects in your ~/work/ directory use the work GitHub account
  • Projects in your ~/projects/ directory use the home GitHub account

One directory cannot contain the other. I use two GitHub accounts: work and home, but the idea supports any number of accounts.

Unsolved

What happens when you clone a repo to ~/some_other_dir? You have to re-do the steps in this article for that directory. What I really want is a setup that does this:

  • Projects in ~/work/ and ~/go use the work GitHub account
  • Projects elsewhere use the home GitHub account

Unfortunately, I haven't gotten that working yet (have you? Contact me!). More about that in the More About Git Config Loading section.

The git config

First, tell git to use different git configurations depending on the directory a project is in. Git supports this with the includeIf directive.

File: ~/.config/git/config 

[includeIf "gitdir:~/projects/"]
    path = home.gitconfig
[includeIf "gitdir:~/work/"]
    path = work.gitconfig

Next, create the home.gitconfig and work.gitconfig files referenced above. Each file specifies an email address and a URL-rewriting rule.

File: ~/.config/git/work.gitconfig 

[user]
    email = cfclrk@work.com
[url "git@github-work"]
    insteadOf = git@github.com
[url "git@gist-work"]
    insteadOf = git@gist.github.com

File: ~/.config/git/home.gitconfig 

[user]
    email = cfclrk@home.com
[url "git@github-home"]
    insteadOf = git@github.com
[url "git@gist-home"]
    insteadOf = git@gist.github.com

That's it for the git config! The rest of this section elaborates on the above, if you are interested.

More About Git URL Rewriting

To take a closer look at what this insteadOf part means: 

[url "git@github-home"]
    insteadOf = git@github.com

Here we've said: "Mr. Git, any time you see a URL that starts with git@github.com, please just pretend like the URL actually starts with git@github-home".

For example, let's see what git thinks about the git repository where I'm writing this article. If I just cat the git config file in my local articles repository, I see the real remote URL: 

cat .git/config | grep "\[remote" -A 2

[remote "origin"]
	url = git@github.com:cfclrk/articles.git
	fetch = +refs/heads/*:refs/remotes/origin/*

But when I use a git command (like git remote) to view that same value, I see the rewritten URL: 

git remote -v

origin	git@github-home:cfclrk/articles.git (fetch)
origin	git@github-home:cfclrk/articles.git (push)

All git commands will see this rewritten form. The rewritten form is determined by whatever git config file was loaded when the git command was run. If we were to simply move this project out of the ~/projects/ directory, a different git config file would be loaded, different URL-rewriting rules would apply, and git remote -v would show a different URL.

The value I chose – in this case "github-home" – could have been any string (well, anything without spaces, etc). It doesn't matter so much what the value is, it just matters that you use also use the same value in your SSH config (as explained in the section The SSH Config).

More About Git Config Loading

Git loads all applicable config files. If a single option is set multiple times (potentially form differnt files), the last value wins.

For example: say you try ~/.config/git/config like this. 

[includeIf "gitdir:/"]
    path = home.gitconfig
[includeIf "gitdir:~/work/"]
    path = work.gitconfig

Now you run a git command in a project located at: ~/work/projectA/.

  • Does [includeIf "gitdir:/"] apply?
    • Yes! Load home.gitconfig
    • Set user.email to cfclrk@home.com
  • Does [includeIf "gitdir:~/work/"] apply?
    • Yes! Load work.gitconfig
    • Set user.email to cfclrk@work.com

In this case, both the work.gitconfig and home.gitconfig are loaded. The email address will end up being set to cfclrk@work.com since that was the last value loaded for the user.email option. The determining factor is the order in which the configuration is defined, and the last value wins.

So, when you define multiple includeIf directives, you should specify the directives in order from most general to most specific.

Aside: The value of user.email is the only piece of information GitHub uses when determining what profile picture to display next to a commit. You can set that email address to anything! E.g. set it to torvalds@linux-foundation.org, and GitHub will happily put Linus Torvalds' picture next to your git commits. And to be sure, that happens. You can partially mitigate this problem of attribution by using signed commits, as discussed in the section Signed Commits.

The SSH Config

Assuming you use the git protocol to clone git repositories (which uses SSH under the covers) and not https like some kind of animal, you have at least two SSH keys for GitHub – one for each GitHub account (GitHub will not allow you to use the same SSH key for both accounts, I've tried). The ~/.ssh/config allows you to specify which SSH key should be used for particular DNS domains. And this is where all tha URL-rewriting stuff comes in to play! We can tell SSH to use one SSH key for github-home, and another SSH key for github-work.

File: ~/.ssh/config 

Host github-home
HostName       github.com
User           git
IdentityFile   ~/.ssh/github-home
IdentitiesOnly yes

Host github-work
HostName       github.com
User           git
IdentityFile   ~/.ssh/github-work
IdentitiesOnly yes

Host gist-home
HostName       gist.github.com
User           git
IdentityFile   ~/.ssh/github-home
IdentitiesOnly yes

Host gist-work
HostName       gist.github.com
User           git
IdentityFile   ~/.ssh/github-work
IdentitiesOnly yes

Signed Commits

In More About Git Config Loading, I mentioned how to make git commits appear to be associated with someone else's GitHub account. To (somewhat) mitigate that situation, GitHub allows you to upload a PGP key to your GitHub account, and GitHub displays a "Verified" badge on commits that are signed with a PGP key that matches the email address associated with the commit.
